01 · Pengantar

Tujuan Pembelajaran & Ringkasan

Autentikasi adalah pintu masuk pertama setiap sistem yang punya nilai. Tanpa autentikasi yang benar, kita tidak bisa membedakan pengguna sah dari penyerang — semua mekanisme keamanan setelahnya runtuh.

Tujuan Pembelajaran

Setelah mengikuti modul ini, mahasiswa diharapkan mampu:

  • Menjelaskan perbedaan autentikasi (siapa kamu) dan otorisasi (apa yang boleh kamu lakukan).
  • Mengenali minimal tujuh jenis autentikasi yang umum dipakai di web dan REST API.
  • Mengaplikasikan aturan dasar keamanan: least privilege, defense in depth, dan secure by default.
  • Memilih metode autentikasi yang tepat berdasarkan kebutuhan aplikasi dan risiko.
  • Mengidentifikasi celah keamanan umum dan menerapkan praktik mitigasi.

Ruang Lingkup Modul

Modul ini dibagi menjadi enam bab utama. Bab 02 menggali alasan mengapa autentikasi diperlukan lewat kerangka 5W1H. Bab 03 memperkenalkan tipe-tipe autentikasi dari yang paling sederhana hingga yang paling modern. Bab 04 menjelaskan aturan umum yang berlaku lintas teknologi. Bab 05 menyajikan tips dan best practice keamanan yang siap pakai. Bab 06 menutup dengan studi kasus nyata yang menghubungkan teori ke implementasi.

Pertanyaan yang dijawab modul ini
Kenapa setiap aplikasi modern memerlukan autentikasi? Bagaimana cara kerja JWT, OAuth, dan API Key — dan kapan masing-masing dipakai? Apa kesalahan paling sering yang membuat sistem autentikasi dapat dibobol?
02 · Landasan

5W1H — Mengapa Butuh Autentikasi

Sebelum membahas bagaimana, kita harus paham kenapa. Kerangka 5W1H (What, Why, Who, When, Where, How) membantu kita membongkar kebutuhan autentikasi secara sistematis.

What — Apa itu Autentikasi?

Autentikasi (authentication) adalah proses memverifikasi bahwa identitas yang diklaim seseorang atau sistem benar adanya. Sederhananya: membuktikan kamu adalah kamu. Hal ini berbeda dengan otorisasi, yang menentukan apa yang boleh dilakukan setelah identitas terverifikasi.

Analogi: di gerbang masuk apartemen, satpam memeriksa KTP — itu autentikasi. Setelah masuk, kartu akses menentukan kamu boleh masuk lift ke lantai berapa — itu otorisasi.

Why — Mengapa Diperlukan?

Setiap aplikasi yang menyimpan data atau memberikan layanan bernilai memerlukan autentikasi karena beberapa alasan fundamental:

  • Melindungi data pribadi pengguna — data medis, finansial, dan komunikasi tidak boleh diakses sembarang orang.
  • Mencegah penyalahgunaan sumber daya — server, kuota API, dan kapasitas komputasi memiliki biaya nyata.
  • Memenuhi kepatuhan regulasi — UU PDP (Indonesia), GDPR (Eropa), HIPAA (kesehatan AS), dan PCI-DSS (pembayaran) mewajibkan kontrol akses yang ketat.
  • Menjaga akuntabilitas dan audit — setiap aksi penting harus bisa dikaitkan dengan identitas yang bertanggung jawab.
  • Membangun kepercayaan pengguna — sistem yang mudah dibobol akan kehilangan reputasi dengan cepat.

Who — Siapa yang Diautentikasi?

Subjek autentikasi tidak selalu manusia. Tiga kategori utama:

  • Pengguna akhir — manusia yang mengakses aplikasi web atau mobile.
  • Layanan / sistem — service-to-service, microservice yang saling memanggil API.
  • Aplikasi pihak ketiga — integrasi via OAuth, bot, atau plugin dari developer eksternal.

When — Kapan Autentikasi Berjalan?

Autentikasi tidak hanya terjadi sekali. Tiga momen penting:

  • Saat login awal — verifikasi penuh dengan username/password atau metode setara.
  • Sepanjang sesi — setiap request membawa bukti (cookie, token) bahwa pengguna masih sah.
  • Saat aksi sensitif — re-authentication untuk transfer dana, ubah password, atau akses data kritikal.

Where — Di Lapisan Mana?

Autentikasi diterapkan di berbagai lapisan arsitektur:

LapisanBentuk AutentikasiContoh
Frontend / BrowserForm login, cookie sessionHalaman login web
API GatewayValidasi token / API keyKong, AWS API Gateway
Aplikasi / ServiceMiddleware autentikasiExpress middleware, Next.js middleware
DatabaseCredential, IAM rolePostgres roles, RDS IAM
InfrastrukturSSH key, mTLSServer access, service mesh

How — Bagaimana Mekanismenya?

Autentikasi bekerja dengan memverifikasi minimal salah satu dari tiga faktor:

  • Something you know — password, PIN, jawaban keamanan.
  • Something you have — ponsel (OTP), token fisik, kartu akses, kunci U2F.
  • Something you are — sidik jari, wajah, suara, iris mata.

Sistem dianggap menggunakan Multi-Factor Authentication (MFA) bila menggabungkan minimal dua faktor yang berbeda kategori. MFA secara dramatis menurunkan risiko pengambilalihan akun.

03 · Tipe Autentikasi

Jenis-Jenis Autentikasi di Web & REST API

Tidak ada satu metode yang cocok untuk semua kasus. Memilih metode autentikasi yang tepat berarti memahami trade-off antara keamanan, kemudahan implementasi, dan pengalaman pengguna.

1. HTTP Basic Authentication

Skema paling sederhana: klien mengirim header Authorization: Basic base64(user:password) di setiap request. Hanya menggunakan Base64 (bukan enkripsi), sehingga wajib digunakan bersama HTTPS.

GET /api/data HTTP/1.1
Authorization: Basic dXNlcjpwYXNzd29yZA==

Pro Cepat untuk diimplementasikan, didukung semua browser dan klien HTTP.
Con Kredensial dikirim di setiap request, sulit logout, tidak ada konsep sesi.

2. Session-Based Authentication (Cookie)

Setelah login berhasil, server membuat session ID dan menyimpannya di server (memory, Redis, atau database). Session ID dikirim ke browser melalui cookie. Setiap request membawa cookie tersebut, dan server memvalidasi sesinya.

Model ini stateful — server harus mengingat setiap sesi. Cocok untuk aplikasi web monolitik tradisional.

3. Token-Based Authentication (JWT)

JSON Web Token adalah token stateless yang membawa informasi pengguna di dalam payload-nya, ditandatangani secara kriptografis. Struktur: header.payload.signature.

eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiIxMjMiLCJleHAiOjE3MDB9.SflKxw...

Server tidak perlu menyimpan sesi — verifikasi cukup dengan memvalidasi tanda tangan. Ideal untuk REST API dan arsitektur microservice. Token dikirim via header Authorization: Bearer <token>.

4. OAuth 2.0 & OpenID Connect

OAuth 2.0 adalah framework otorisasi yang memungkinkan aplikasi pihak ketiga mengakses sumber daya pengguna tanpa membagikan password. OpenID Connect (OIDC) menambahkan lapisan autentikasi di atas OAuth 2.0.

Empat alur (grant types) utama:

  • Authorization Code — untuk aplikasi web server-side (paling aman).
  • Authorization Code + PKCE — untuk SPA dan mobile (versi modern).
  • Client Credentials — untuk komunikasi service-to-service.
  • Device Code — untuk perangkat tanpa keyboard (smart TV, IoT).

5. API Key

String panjang yang unik per klien atau aplikasi, dikirim via header (umumnya X-API-Key) atau query parameter. Sederhana, cocok untuk akses program-ke-program di mana identitas pengguna tidak penting.

Contoh layanan yang memakai: Stripe, OpenAI, Google Maps API, SendGrid.

6. Multi-Factor Authentication (MFA / 2FA)

Bukan metode autentikasi tunggal, tapi lapisan tambahan. Pengguna harus memberikan minimal dua bukti dari kategori berbeda. Implementasi umum:

  • TOTP — Google Authenticator, Authy (kode 6 digit yang berubah tiap 30 detik).
  • SMS OTP — rentan SIM swapping, tidak direkomendasikan untuk akun bernilai tinggi.
  • Push notification — Duo, Microsoft Authenticator.
  • Hardware key — YubiKey, Google Titan (paling kuat secara kriptografis).

7. SAML & Single Sign-On (SSO)

Security Assertion Markup Language adalah standar berbasis XML untuk SSO di lingkungan enterprise. Identitas dikelola oleh Identity Provider (IdP) seperti Okta, Azure AD, atau Google Workspace. Pengguna login sekali, dapat akses ke banyak aplikasi.

8. Mutual TLS (mTLS)

Berbeda dari HTTPS biasa yang hanya memverifikasi server, mTLS juga memverifikasi klien menggunakan sertifikat X.509. Standar de facto untuk komunikasi service-to-service di service mesh dan zero-trust network.

9. Passwordless & WebAuthn / Passkeys

Tren modern: menghilangkan password sama sekali. WebAuthn (Web Authentication API) memungkinkan login pakai biometrik perangkat atau hardware key. Passkeys (Apple, Google, Microsoft) adalah penerapan WebAuthn yang sinkron lintas perangkat. Tahan terhadap phishing karena kunci tidak pernah meninggalkan perangkat.

04 · Aturan Dasar

Aturan Umum Autentikasi & Otorisasi

Tanpa memahami prinsip dasar di balik autentikasi dan otorisasi, kita rentan membuat keputusan implementasi yang tampak benar tapi sebetulnya membuka celah. Bab ini merangkum aturan yang berlaku lintas teknologi.

AuthN vs AuthZ — Dua Hal yang Berbeda

Salah satu kesalahan paling umum adalah mencampuradukkan keduanya. Pisahkan secara tegas:

AspekAutentikasi (AuthN)Otorisasi (AuthZ)
PertanyaanSiapa kamu?Apa yang boleh kamu lakukan?
TerjadiSebelumSetelah autentikasi
OutputIdentitas terverifikasiIzin / penolakan
ContohLogin dengan email/passwordHanya admin yang bisa hapus user

Principle of Least Privilege

Berikan hak akses seminimal mungkin yang dibutuhkan untuk menjalankan tugas. Pengguna baru tidak otomatis admin. Service yang hanya membaca data tidak perlu hak menulis. Ini membatasi blast radius ketika sebuah akun atau service terbobol.

Defense in Depth

Jangan mengandalkan satu lapisan saja. Bila WAF terlewat, ada rate limiting. Bila rate limiting terlewat, ada validasi input. Bila validasi terlewat, ada parameterized query. Setiap lapisan harus berasumsi lapisan di atasnya bisa gagal.

Secure by Default

Konfigurasi paling aman harus menjadi default. Cookie harus HttpOnly dan Secure tanpa perlu diatur. Endpoint baru harus memerlukan autentikasi kecuali eksplisit di-allow-list publik. Lebih baik error karena terlalu ketat, daripada bocor karena terlalu longgar.

Zero Trust

Asumsikan tidak ada zona yang inheren aman. Bahkan internal network bisa dikompromikan. Setiap request — termasuk service-to-service — harus diautentikasi dan diotorisasi. Slogan: never trust, always verify.

Don't Roll Your Own Crypto

Jangan menciptakan algoritma kriptografi atau skema hashing sendiri. Gunakan library yang sudah teruji: bcrypt, argon2, scrypt untuk password; HMAC-SHA256 atau library JWT yang well-known untuk token.

Aturan praktis: kalau Anda merasa kreatif dengan keamanan, berhenti dulu. Konsultasikan ke kolega senior atau review komunitas. Implementasi kriptografi yang salah tampak baik-baik saja sampai dibobol.

Fail Securely

Saat terjadi error, sistem harus gagal ke arah yang lebih aman, bukan lebih terbuka. Bila pemeriksaan otorisasi melempar exception, default harus menolak akses — bukan mengizinkannya.

Separation of Concerns

Pisahkan logic autentikasi dan otorisasi dari logic bisnis. Tempatkan di middleware terpusat. Kalau setiap controller punya implementasi cek-izin sendiri, satu controller yang lupa cek akan menjadi celah.

Audit Trail

Setiap aksi penting — login, perubahan password, akses data sensitif, perubahan izin — harus tercatat dengan informasi: siapa, kapan, dari mana (IP, user agent), apa yang dilakukan. Audit log harus immutable dan disimpan terpisah dari database utama.

05 · Best Practice

Security Tips & Best Practice

Bab ini berisi praktik konkret yang siap diterapkan — daftar yang harus dicentang sebelum aplikasi Anda dianggap layak produksi.

Penyimpanan Password

  • Jangan pernah simpan password plaintext. Pernah ada perusahaan besar yang melakukannya — dan terbongkar saat database bocor.
  • Gunakan algoritma password hashing yang dirancang khusus: bcrypt, argon2id (paling direkomendasikan), atau scrypt. Hindari MD5, SHA-1, atau SHA-256 polos — terlalu cepat dan rentan brute force.
  • Setiap password harus pakai salt unik. Library modern menangani ini otomatis.
  • Atur work factor sehingga hashing memakan ~250 ms di server produksi — cukup lambat untuk menghambat brute force, cukup cepat untuk UX login.

Manajemen Sesi & Token

  • Session ID harus random dan minimal 128 bit entropy.
  • Cookie sesi: HttpOnly (tidak bisa dibaca JavaScript), Secure (hanya via HTTPS), SameSite=Lax atau Strict (proteksi CSRF).
  • Set expiry yang wajar — JWT access token: 15 menit hingga 1 jam. Refresh token: hari hingga minggu, rotasi setiap pemakaian.
  • Invalidasi sesi saat logout, ganti password, atau aktivitas mencurigakan.
  • Untuk JWT: simpan blacklist untuk token yang harus di-revoke segera (atau pakai opaque token).

Proteksi terhadap Brute Force

  • Rate limiting per IP dan per akun — misal maksimum 5 percobaan login per menit.
  • Account lockout sementara setelah N percobaan gagal — tapi hati-hati, ini bisa dipakai untuk DoS akun pengguna.
  • CAPTCHA atau proof-of-work setelah beberapa percobaan gagal.
  • Pantau pola credential stuffing — banyak percobaan dengan kredensial bocor dari breach lain.

Komunikasi Aman

  • HTTPS di mana-mana — tidak ada pengecualian, bahkan di staging. Pakai HSTS untuk memaksa.
  • TLS minimal versi 1.2, idealnya 1.3. Disable cipher lama (RC4, 3DES).
  • Sertifikat dari Let's Encrypt atau CA tepercaya. Otomatisasi renewal.

Proteksi Lapisan Web

  • CSRF — pakai token CSRF, SameSite cookie, atau header verification untuk endpoint state-changing.
  • XSS — escape semua output, gunakan Content Security Policy (CSP), hindari dangerouslySetInnerHTML.
  • Clickjacking — set header X-Frame-Options: DENY atau CSP frame-ancestors 'none'.
  • Open redirect — validasi URL tujuan setelah login, jangan trust parameter mentah.

Validasi Input & Pesan Error

  • Validasi semua input di sisi server, bukan hanya di klien.
  • Pesan error login harus generic — "Email atau password salah", bukan "Password salah" (yang membocorkan bahwa email ada).
  • Jangan bocorkan stack trace atau detail teknis ke pengguna akhir.

Multi-Factor Authentication

  • Wajibkan MFA untuk akun admin, akun developer, dan akses ke data sensitif.
  • Tawarkan MFA opsional untuk semua pengguna.
  • Prioritas: hardware key > push notification > TOTP > SMS (SMS hanya sebagai fallback terakhir).
  • Sediakan recovery code sekali pakai untuk pemulihan akun.

Secret Management

  • Jangan commit credential ke git. Pakai .env yang di-gitignore, atau secret manager (AWS Secrets Manager, HashiCorp Vault, Doppler).
  • Rotasi secret secara berkala, terutama saat ada developer keluar tim.
  • Pakai variabel environment untuk konfigurasi sensitif, bukan hardcode.

Logging & Monitoring

  • Catat: login berhasil/gagal, perubahan password, perubahan email, perubahan izin, akses data sensitif.
  • Jangan log password, token utuh, atau data sensitif lainnya.
  • Set alerting untuk pola anomali — login dari geografi baru, banyak percobaan gagal, akses di luar jam normal.
Checklist Pre-Production
HTTPS aktif? Password di-hash dengan argon2/bcrypt? Rate limiting di endpoint login? Cookie HttpOnly+Secure+SameSite? Token punya expiry? MFA tersedia? Pesan error tidak membocorkan info? Audit log berjalan? Secret tidak hardcode? Bila semua centang — Anda sudah lebih aman dari mayoritas aplikasi di internet.
06 · Studi Kasus

Use Case Nyata per Tipe Autentikasi

Teori menjadi hidup ketika dikaitkan dengan implementasi nyata. Bab ini memetakan setiap tipe autentikasi ke skenario industri yang sebenarnya — sehingga ketika Anda bertemu kebutuhan serupa, Anda tahu harus mulai dari mana.

Tipe — Basic Authentication

Dashboard Internal Tim Operasional

Konteks: sebuah startup butuh halaman admin sederhana untuk lima orang tim operasional. Halaman ini hanya diakses dari kantor.

Kenapa cocok: jumlah pengguna sangat sedikit, tidak ada kebutuhan logout/refresh token, dan implementasi cukup beberapa baris kode di reverse proxy (Nginx, Caddy). Cepat dipasang, cukup aman bila digabung dengan IP allowlist dan HTTPS.

Kapan jangan dipakai: aplikasi yang dihadapkan ke publik, atau yang menyimpan data pengguna lain.

Tipe — Session Cookie

E-commerce Tradisional (Laravel, Rails, Django)

Konteks: toko online dengan keranjang belanja, riwayat pesanan, alamat pengiriman. Pengguna login lalu melakukan banyak aksi dalam satu sesi.

Kenapa cocok: framework klasik sudah menyediakan session management matang. Server-side state memudahkan logout instan, deteksi multi-device, dan invalidasi sesi saat ada perubahan password. Pengalaman pengguna mulus karena cookie ter-handle browser.

Catatan: simpan session di Redis (bukan memory) supaya horizontal scaling tetap mudah.

Tipe — JWT (JSON Web Token)

Aplikasi Mobile + REST API Backend

Konteks: aplikasi ride-hailing dengan klien Android, iOS, dan dashboard web. Backend microservice (auth service, ride service, payment service) saling terpisah.

Kenapa cocok: JWT stateless — payment service bisa memverifikasi token sendiri tanpa memanggil auth service untuk setiap request. Ini mengurangi latency dan menghilangkan single point of failure. Mobile app menyimpan token di Keychain/Keystore, dan refresh token-nya secara berkala.

Pitfall yang harus dihindari: jangan simpan JWT di localStorage (rentan XSS) — pakai cookie HttpOnly atau memory untuk SPA web. Selalu validasi exp dan signature.

Tipe — OAuth 2.0 / OpenID Connect

"Login dengan Google" di Aplikasi SaaS

Konteks: SaaS produktivitas (mirip Notion atau Figma) ingin onboarding tanpa friksi. Pengguna malas membuat password baru.

Kenapa cocok: Google/Microsoft sudah menangani password security, MFA, dan recovery. Aplikasi cukup percaya identitas yang divalidasi oleh provider. OpenID Connect memberikan ID token berisi informasi pengguna (nama, email, foto).

Bonus use case: integrasi pihak ketiga — aplikasi Anda ingin baca kalender pengguna di Google Calendar tanpa minta password Google. OAuth 2.0 menyelesaikannya dengan scoped access token.

Tipe — API Key

Layanan Pembayaran ke Sistem Merchant

Konteks: Midtrans, Xendit, atau Stripe memberikan API kepada merchant untuk membuat transaksi pembayaran. Yang mengakses adalah server merchant, bukan pengguna akhir.

Kenapa cocok: tidak ada konsep "pengguna" dalam interaksi ini — yang penting adalah merchant X yang sah memanggil API. API key dengan rate limit per kunci, scope terbatas, dan bisa di-revoke independen sangat cocok.

Best practice: beri test key (sandbox) dan live key terpisah. Tampilkan key sekali saat dibuat, hash-kan di database. Sediakan UI untuk rotasi.

Tipe — MFA / 2FA

Internet Banking & Dompet Digital

Konteks: aplikasi bank mobile atau e-wallet menyimpan dana nyata pengguna. Kebocoran password tunggal tidak boleh berakibat pencurian dana.

Kenapa cocok: regulasi (Bank Indonesia, OJK) mensyaratkan strong customer authentication. MFA dengan PIN + OTP (push ke aplikasi) + biometrik perangkat membuat pengambilalihan akun jauh lebih sulit.

Pola umum: login awal dengan PIN, transaksi besar memerlukan re-authentication via biometrik atau OTP. Setiap perangkat baru harus diverifikasi.

Tipe — SAML / SSO

Korporasi 5000 Karyawan dengan 30+ SaaS

Konteks: perusahaan memakai Slack, Jira, GitHub, Salesforce, Notion. Setiap karyawan harus login ke semuanya — dan ketika seseorang resign, akses harus dicabut serempak.

Kenapa cocok: SAML memungkinkan IdP (Okta, Azure AD) menjadi sumber kebenaran identitas. Onboarding sekali klik, offboarding sekali klik. Audit log terpusat. MFA cukup di IdP, semua aplikasi ikut.

Catatan: SAML kompleks dan berbasis XML. Untuk integrasi modern, OIDC sering lebih sederhana dan disukai vendor baru.

Tipe — Mutual TLS (mTLS)

Komunikasi Antar Microservice di Service Mesh

Konteks: arsitektur dengan puluhan microservice yang saling memanggil di cluster Kubernetes. Setiap service harus yakin bahwa pemanggilnya benar service yang dimaksud, bukan attacker yang menembus internal network.

Kenapa cocok: mTLS memverifikasi kedua sisi dengan sertifikat. Istio / Linkerd menangani penerbitan dan rotasi sertifikat otomatis. Tidak ada token yang bisa bocor — kompromi membutuhkan kunci privat.

Use case lain: integrasi B2B finansial yang diatur regulasi (open banking, host-to-host).

Tipe — WebAuthn / Passkeys

Login Modern di Google, Apple, GitHub

Konteks: platform besar ingin menghilangkan password yang menjadi sumber phishing dan kebocoran. Passkey memungkinkan login pakai biometrik perangkat (Face ID, sidik jari) dengan kunci kriptografis tersinkronisasi via cloud akun.

Kenapa cocok: tahan phishing (kunci terikat ke origin asli), tidak ada yang bisa dicuri dari database (server hanya menyimpan public key), UX jauh lebih cepat daripada password + OTP.

Tren: sebagian besar platform besar sekarang mendukung passkey sebagai alternatif. Ekspektasinya, dalam 5-10 tahun, password akan menjadi metode legacy.

Pola Pemilihan Cepat

Bila ragu memilih metode, gunakan pohon keputusan berikut:

  • Aplikasi web tradisional, satu domain → Session Cookie.
  • SPA atau mobile + REST API → JWT (access + refresh token).
  • Butuh "Login dengan X" → OAuth 2.0 / OIDC.
  • API untuk developer eksternal → API Key.
  • Komunikasi service-to-service internal → mTLS.
  • Akun bernilai tinggi (uang, data medis, akses produksi) → tambahkan MFA.
  • Konteks korporasi dengan banyak SaaS → SAML / OIDC SSO.
07 · Lampiran

Glosarium & Bacaan Lanjutan

A. Glosarium

Autentikasi (AuthN) — proses verifikasi identitas: membuktikan bahwa subjek adalah siapa yang diklaim.

Otorisasi (AuthZ) — proses menentukan apa yang boleh dilakukan oleh identitas yang sudah terverifikasi.

Bearer Token — token yang dibawa di header Authorization; siapa yang memegang token, dia dianggap pemilik sah.

JWT (JSON Web Token) — format token compact dan self-contained yang ditandatangani secara kriptografis.

OAuth 2.0 — framework otorisasi untuk delegasi akses ke sumber daya tanpa membagikan kredensial.

OpenID Connect (OIDC) — lapisan autentikasi yang dibangun di atas OAuth 2.0.

SAML — Security Assertion Markup Language, standar SSO berbasis XML yang dominan di enterprise.

MFA / 2FA — Multi-Factor / Two-Factor Authentication, menggabungkan dua atau lebih faktor identitas.

TOTP — Time-based One-Time Password, kode 6 digit yang berubah tiap 30 detik (Google Authenticator).

mTLS — Mutual TLS, autentikasi dua arah menggunakan sertifikat X.509.

WebAuthn — standar W3C untuk autentikasi berbasis kriptografi kunci publik, dasar dari Passkey.

Salt — nilai acak unik yang digabung dengan password sebelum hashing untuk mencegah rainbow table.

CSRF — Cross-Site Request Forgery, serangan yang memanfaatkan sesi pengguna untuk aksi tak diinginkan.

XSS — Cross-Site Scripting, injeksi script berbahaya ke halaman yang dilihat pengguna lain.

Zero Trust — paradigma keamanan yang tidak mempercayai zona apa pun; setiap request harus diverifikasi.

B. Bacaan Lanjutan

  • OWASP Authentication Cheat Sheet — referensi praktis dari komunitas keamanan terbuka.
  • OWASP Top 10 — daftar kerentanan paling kritis di aplikasi web, diperbarui berkala.
  • RFC 6749 — spesifikasi resmi OAuth 2.0.
  • RFC 7519 — spesifikasi resmi JSON Web Token.
  • NIST SP 800-63B — pedoman digital identity dari pemerintah AS, sumber otoritatif tentang authenticator.
  • "Pragmatic Web Security" oleh Philippe De Ryck — kursus dan artikel mendalam.
  • auth0.com/docs dan okta.com/identity-101 — dokumentasi vendor yang sangat edukatif.

C. Latihan untuk Mahasiswa

  1. Pilih satu aplikasi yang Anda pakai sehari-hari. Identifikasi: tipe autentikasi apa yang dipakai? MFA tersedia? Bagaimana Anda mengetahuinya?
  2. Implementasikan login sederhana dengan JWT pada Node.js/Express atau Next.js. Buat endpoint /login, /refresh, dan /me.
  3. Buat tabel perbandingan tiga metode autentikasi untuk skenario: "platform forum diskusi dengan 100 ribu pengguna". Argumentasikan pilihan terbaik.
  4. Audit aplikasi web open source pilihan Anda. Cari kerentanan: apakah cookie pakai HttpOnly? Apakah ada rate limit? Pesan error login generic?
